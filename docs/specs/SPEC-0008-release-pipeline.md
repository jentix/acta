# Summary

The release pipeline turns the completed MVP into a publishable project. It defines which packages can be published to npm, how version and changelog changes are prepared, and how the repository's own Acta documents are deployed as a static dogfooding demo on GitHub Pages.

# Goals

- Publish `@acta/cli` with a working `acta` binary and `@acta/core` for integrations.
- Keep `@acta/renderer` and `@acta/web` private for the MVP because they are implementation details of the current viewer.
- Use Changesets as the single workflow for package version bumps, changelog entries and publish commands.
- Deploy the static web viewer from the repository's own ADR/spec corpus on pushes to `main`.
- Make the release contract testable in CI so package metadata and deployment configuration do not silently regress.

# Requirements

- The root package exposes `pnpm changeset`, `pnpm version-packages` and `pnpm release` scripts.
- `packages/cli/package.json` includes `bin.acta`, `files`, `engines` and public npm `publishConfig`.
- `packages/core/package.json` includes `files`, `engines` and public npm `publishConfig`.
- `packages/renderer/package.json` and `apps/web/package.json` remain private.
- `.changeset/config.json` is configured for public packages, `main` as the base branch and patch updates for internal dependencies.
- `CHANGELOG.md` records the MVP release summary.
- `apps/web/astro.config.mjs` sets the GitHub Pages `site` and repository `base` path.
- `.github/workflows/deploy-pages.yml` builds the CLI, builds Acta artifacts, builds `@acta/web` and uploads `apps/web/dist` through GitHub Pages actions.
- The repository keeps a Vitest release-contract test that verifies these release-critical files.

# Proposed design

Changesets owns the package release workflow. For the MVP, `@acta/cli` and `@acta/core` are the only publishable packages. The CLI gives users the executable workflow, and core gives future consumers the stable document pipeline and artifact contract. The renderer and web app are kept private to avoid committing to their package APIs before the viewer design settles.

GitHub Pages deployment is separate from npm publishing. The Pages workflow runs only on `main` and can also be triggered manually. It builds the CLI first because the static viewer consumes artifacts generated by `acta build`; then it builds Astro with `base: "/adr-book"` and uploads `apps/web/dist`.

Release safety is covered by `packages/cli/src/release-contract.test.ts`. The test reads package metadata, Changesets config, Astro config and the Pages workflow as files, which keeps the release contract close to the codebase and catches accidental removal of required fields.

# Open questions

- Whether future releases should add a dedicated GitHub Actions workflow for `changeset version` pull requests.
- Whether `@acta/renderer` should become public once the web viewer API stabilizes.
