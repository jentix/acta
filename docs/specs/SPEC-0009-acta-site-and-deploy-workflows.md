# Summary

Phase 3 makes the Acta web viewer deployable from any repository that contains Acta documents. The user runs `acta site` to build a static viewer from `.acta/dist`, or runs `acta init --deploy=<provider>` to scaffold a GitHub Actions workflow that builds and publishes `.acta/site` without cloning the Acta monorepo.

# Goals

- Let users publish their own ADR/spec viewer with one CLI flag and a commit.
- Keep `@acta-dev/core` artifacts as the viewer's data contract.
- Decouple the published viewer from the Acta monorepo checkout.
- Make local preview and CI deployment use the same generated site output.
- Provide GitHub Pages as the fully supported deployment path and scaffold common static hosts.

# Requirements

- The web viewer must read generated artifacts from `.acta/dist` instead of loading Markdown directly in deployed builds.
- `acta site` must run the artifact build, build the published `@acta-dev/web` viewer, and write static output to `.acta/site` by default.
- `acta site` must support `--out`, `--base`, `--site`, `--skip-build`, `--config`, and `--json`.
- `acta site --serve` must serve the generated static output locally over HTTP until the process is stopped.
- `acta site --serve` must support `--host` and `--port`, handle directory `index.html`, set useful MIME types, and reject path traversal.
- `acta site --serve` must be incompatible with `--json`.
- The `site` block in `acta.config.ts` must allow default `outDir`, `base`, and `url` values.
- `@acta-dev/web` and `@acta-dev/renderer` must be publishable packages so the CLI can build the viewer in a user's repository.
- `acta init --deploy=pages` must write a GitHub Pages workflow that runs `pnpm dlx @acta-dev/cli site --base "/${{ github.event.repository.name }}"`, uploads `.acta/site`, and deploys with `actions/deploy-pages`.
- `acta init --deploy=cloudflare|vercel|netlify` must write scaffold workflows that build `.acta/site` and include the required provider secrets/placeholders.
- Invalid deploy providers must exit with a CLI usage error.
- Documentation must explain how to publish the viewer in a user's own CI and which provider secrets are required.

# Proposed design

The deployable viewer uses `.acta/dist` as its runtime input. `apps/web` resolves artifacts through environment-driven paths: `ACTA_DIST_DIR` from `acta site` first, then development fallbacks for monorepo workflows. This keeps Markdown parsing, validation, graph building, ordering, and search index generation inside `@acta-dev/core`.

`acta site` is the build bridge between a user's repository and the published viewer package. It resolves the user's Acta config, refreshes artifacts with `buildArtifacts`, resolves `@acta-dev/web` and Astro, then runs the viewer build with `ACTA_PROJECT_ROOT`, `ACTA_DIST_DIR`, `ACTA_SITE_OUT`, `ACTA_BASE`, and `ACTA_SITE`. The generated `.acta/site` directory is the only deploy artifact users need to publish.

Local preview is a mode of the same command instead of a separate development server. `acta site --serve` builds the static site, serves files from the output directory, respects the configured base path, and keeps running until interrupted. Machine-readable output stays reserved for non-long-running `--json` runs.

Deploy workflows are generated by `acta init` because they are repository scaffolding, like hooks and CI templates. GitHub Pages is the primary end-to-end path. Cloudflare Pages, Vercel, and Netlify are generated as host-specific scaffolds with explicit secrets and variables because final deployment details depend on the user's provider account.

# Acceptance criteria

- `acta site` builds `.acta/dist` and `.acta/site` in this repository.
- `acta site --serve` serves the generated site locally and rejects `--json`.
- `acta init --deploy=pages` creates `.github/workflows/acta-deploy-pages.yml`.
- The Pages workflow builds with the published CLI, uploads `.acta/site`, and deploys through GitHub Pages actions.
- `acta init --deploy=cloudflare`, `acta init --deploy=vercel`, and `acta init --deploy=netlify` create provider workflows with documented secret names.
- `acta init --deploy=<unknown>` exits with usage code `2`.
- CLI tests cover site option resolution, static serving, deploy workflow generation, invalid deploy providers, and overwrite behavior.
- Workspace lint, typecheck, tests, build, `acta validate`, and `acta build` pass.

# Open questions

- Whether future releases should add real end-to-end smoke tests against Cloudflare Pages, Vercel, or Netlify.
- Whether the published viewer package should eventually expose a stable public API, or remain an implementation dependency of `acta site`.
